1 — High level architecture (POC-friendly)
Frontend → POST /api/feedback/sentence or /api/feedback/draft

Backend Router → Orchestrates pipelines (Sentence pipeline, Group pipeline, Draft pipeline)

In‑memory Vector Store (FAISS / Annoy) built at start from legal corpora (statutes, commonly-cited sections, RERA/IPC extracts, sample judgments). Keeps POC DB‑less (on-disk JSON + embeddings loaded into RAM).

LLM Host → Ollama running llama3.2:3b-instruct-fp16

Verifier → small entailment / classifier model (tiny Llama or DistilBERT-like) to check claim ↔ retrieved evidence

Search Interface → Open‑source web search tool (scraper + SERP lib) to fetch authoritative sources if in doubt

Cache layer → Redis or in-memory LRU (for POC, simple in-memory dict) to reuse recent retrievals/answers

Scoring & Decision Engine → combines generator confidence, verifier score, and retrieval support → final verdict

2 — Pipeline: sentence-level feedback (efficient + minimal usage)
Goal: give feedback for each sentence with minimal calls.

Debounce + Local segmentation

Receive full editor content but only evaluate last complete sentence (detect by . ? ! \n).

Wait 800–1200ms of inactivity (frontend debounced) — reduces calls.

Lightweight retrieval

Create a short query from the sentence (extract keywords, statutes, sections).

Run vector search (top K=3) against local legal corpus (embeddings precomputed).

If top match score > R_high (e.g., cosine > 0.78), mark as strongly grounded.

Run quick verifier (cheap)

Input: sentence + retrieved snippets

Verifier model returns entailment_score ∈ [0,1] — single small model inference (cheaper than main LLM).

If entailment_score > V_good (0.85), mark claim supported.

If supported → small LLM call (single-turn)

Prompt the Llama model with:

system: “You are a terse legal drafting assistant.”

user: sentence + 1–2 retrieved snippets + instruction: “Return JSON: clarity, issue (if any), improved_sentence (single-line), confidence 0–1.”

This is a single, small response (constrained tokens).

If unsupported or uncertain

Skip heavy generation; instead return: “Claim not supported by available sources — please verify. Sources: [links].” and an improvement suggestion generated by a cheap paraphraser (smaller model or deterministic rules).

Why this saves tokens: verifier + retrieval often avoid calling the big model. If Llama is called, it’s a constrained single-turn answer.

3 — Pipeline: grouped sentence checks (2–5 sentence blocks)
Some legal points only make sense across multiple sentences (e.g., chronology, cause of action). Treat groups similarly but with slightly different thresholds.

Grouping heuristics

Group paragraphs or N sentences (configurable N=3 by default) or on blank line.

Frontend should also allow “Check selection” for arbitrary groups.

Retrieve broader context

Query vector store with combined keywords + legal concepts extracted from the block.

Increase K to 5 to capture more evidence.

Two-step verification

(A) Verifier entailment per claim (extract factual claims from block using a tiny NER/claim-extractor)

(B) Structural check via Llama: ask for missing legal elements, chronology errors, legal theory gaps.

For Llama calls: adopt a structured prompt asking for JSON output (issues[], missing_sections[], suggested_rewrite[]). Limit output tokens.

Aggregate decisions

Use conservative merging: if any claim lacks support and is critical (legal term/section/IPC), mark the entire block as “Needs verification.”

4 — Draft-level evaluation (entire document)
Use only when user clicks “Get AI Feedback” (not on every keystroke).

Full retrieval

Extract all claims, statutes, dates, sections, names.

Run vector search + web search for claims flagged by extractor.

Run Draft LLM prompt

Provide structure: question, full draft, top retrieved evidence (3–5 snippets), and explicit instruction to annotate draft inline with suggested fixes and a final score 0–10.

Ask Llama to output:

issues[] — high-level

annotatedDraft — with inline tags (e.g., [MISSING_PRAYER], [UNCITED_CLAIM])

score — integer

confidence — 0–1

Cross‑verify statutory claims

For every [SECTION: X] or explicit statutory citation, run verifier + retrieval for that section text. If mismatch, flag as potential hallucination.

Return unified payload to frontend: sentenceFeedback[] + draftFeedback + factCheck[]

5 — Fact verification algorithm (practical)
Goal: detect factual/statutory hallucination and reduce false positives.

Claim extraction

Use a small extractor (rules + regex for section patterns like Section 420, dates, judgments).

For each claim record {"claimText", "type": "statute|date|fact|judgment", "anchorSentenceIndex"}

Retrieve sources

Query vector store for claimText

If low confidence, run open-source web search (scraper) using query variations.

Entailment

Pass (claimText, retrieved_snippet) to entailment verifier → get p_entail.

If p_entail >= 0.85 → SUPPORTED

If 0.5 <= p_entail < 0.85 → PARTIAL / needs human check

If p_entail < 0.5 → NOT SUPPORTED

Evidence aggregation

Keep support_count and avg_score. Also include the source URLs/snippets.

If NOT SUPPORTED

Mark sentence/block as “UNVERIFIED”

Provide recommended rewrite: remove the unsupported factual claim OR rephrase as “alleged” / “as per client” rather than asserting as fact.

6 — Prompt engineering principles (for Llama)
Always require JSON only outputs for parsability.

Provide retrieved evidence in the prompt (short snippets).

Use explicit instructions to avoid chain-of-thought and hallucination, e.g.:

System: You are a concise legal drafting assistant. ALWAYS produce only valid JSON. DO NOT invent case law or statutes. If evidence for a claim isn't provided in the 'EVIDENCE' section, mark it "UNVERIFIED".
User: EVIDENCE: [snippet1, snippet2...]
       DRAFT: "..."
       TASK: For each paragraph, return JSON with fields: issues[], suggestions[], rewrittenParagraph, citedEvidenceIndices[], confidence.
Keep token budgets small: restrict max_tokens per response.

7 — Efficiency & cost controls (practical)
Debounce + only evaluate last sentence — reduces calls dramatically.

Verifier-first: run cheap entailment & retrieval; only escalate to Llama when necessary.

Use smaller LLM prompt sizes and max_tokens caps.

Batch processing: when user asks draft feedback, batch sentence checks into single Llama call (reduce per-sentence overhead).

Caching: cache retrieval results and entailment outcomes for identical claims.

Adjustable confidence thresholds (make them configurable).

8 — Handling hallucinations (operational rules)
Require evidence in prompt. If evidence missing, model must respond “UNVERIFIED” not invent.

Use explicit refusal style for unsupported legal assertions.

Use two-model consensus: main Llama generates improvement; small verifier confirms whether that improvement introduced new facts. If verifier flags, reject and ask for human review.

For legal sections: always show the exact statute text or authoritative URL if model cites a section. If model outputs a statute without source, mark low confidence.

9 — Monitoring, metrics & thresholds
Track these to iterate:

calls_per_minute (frontend → backend)

LLM_calls_saved_rate (how often verifier avoided Llama)

unsupported_claim_rate (claims with p_entail < 0.5)

user_acceptance_rate (how often users accept suggested rewrites)

latency (ms)
Log flagged hallucinations and review manually to refine retrieval corpus and prompts.
